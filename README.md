## 6월 파트별 네트워킹 (1) 컴포넌트 단위 과제 가이드라인 및 PR 작성 표준



### 1\.  목적



단일 페이지에 복합적으로 모여있는 App 코드를 여러 컴포넌트와 커스텀 훅으로 분리하여 가독성 향상과 유지보수와 확장을 용이하게 만드는 것을 목적으로 함 



  * **컴포넌트 아키텍쳐 설계**: 논리적이고 재사용 가능한 컴포넌트 계층 구조로 분해하기 

  * **커스텀 훅 설계**: 서로 다른 데이터를 한 커스텀 훅으로 관리하는게 아니라 각각의 단일 책임을 가질 수 있도록 분해하기  

  * **전역 상태 관리**: 다크 모드와 라이트 모드 기능을 지원함으로 해당 기능에 대한 전역 상태관리 해보기 (Context API, zustand 등 원하는 방식)

  * **상태 간의 의존성 관리**: 하나의 상태가 다른 상태에 영향을 미치는 의존성이 있는지 파악해보고 효과적으로 처리해보기   



-----

### 2. 기능 요구사항 



1. **프로젝트 목록**: API에서 프로젝트 목록을 가져와 표시하기. 프로젝트를 클릭하면 해당 프로젝트가 '선택' 상태가 되어야 함



2. **할 일 목록**: 선택된 프로젝트에 속한 할 일(Task) 목록을 API에서 가져와 표시하기



3. **할 일 필터링 및 검색**:

   - 상태(To Do, In Progress, Done)별로 할 일 목록을 필터링할 수 있어야 함

   - 할 일 제목으로 목록을 검색할 수 있어야 함



4. **할 일 추가**: 선택된 프로젝트에 새로운 할 일을 추가하는 폼을 제공하기. 이 폼에서는 할 일의 담당자를 전체 사용자 목록에서 선택할 수 있어야 함



5. **할 일 상태 변경**: 각 할 일 항목에서 상태를 변경할 수 있는 드롭다운 메뉴를 제공하기



6. **테마 변경**: 다크/라이트 모드 토글 시 애플리케이션의 전반적인 색상이 변경되어야 함







### 3. API 간단 요약 (MSW 활용)

(MSW: 실제 API 서버 없이 프론트엔드 개발 및 테스트를 가능하게 하는 라이브러리)

---



#### 1. 프로젝트 목록 조회

* **경로**: `GET /api/projects`

* **파라미터**: 없음

* **본문**: 없음

* **응답**: 전체 프로젝트 목록 (배열)



---



#### 2. 사용자 목록 조회

* **경로**: `GET /api/users`

* **파라미터**: 없음

* **본문**: 없음

* **응답**: 전체 사용자 목록 (배열)



---



#### 3. 특정 프로젝트의 작업 목록 조회

* **경로**: `GET /api/tasks`

* **파라미터**: `projectId` (쿼리 스트링)

* **본문**: 없음

* **응답**: 해당 프로젝트의 작업 목록 (배열)



---



#### 4. 새로운 작업 생성

* **경로**: `POST /api/tasks`

* **파라미터**: 없음

* **본문**: 생성할 작업 정보 (JSON)

* **응답**: 생성된 작업 정보 (JSON)



---



#### 5. 작업 상태 수정

* **경로**: `PATCH /api/tasks/:taskId`

* **파라미터**: `taskId` (URL 경로)

* **본문**: 수정할 상태 정보 (JSON)

* **응답**: 수정된 작업 정보 (JSON)



-----
